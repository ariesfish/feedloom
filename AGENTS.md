# AGENTS.md

This file provides guidance for AI coding agents working on this repository.

## Project Overview

Feedloom is a TypeScript CLI tool for archiving long-form web content as clean Markdown with local assets. It accepts article URLs, URL list files, or RSS/Atom feeds, fetches the pages, extracts readable content, converts it to Markdown, and writes notes with YAML frontmatter.

The package is an ESM Node.js CLI targeting Node.js 24+.

## Tech Stack

- Runtime: Node.js >= 24
- Language: TypeScript
- Module system: ESM (`"type": "module"`)
- Package manager: npm
- CLI framework: Commander
- Build: tsup
- Type checking: TypeScript
- Tests: Vitest
- Browser automation: Patchright
- HTML/content extraction: Defuddle, Linkedom, custom cleaning/site rules
- Markdown rendering: Turndown + GFM plugin

## Common Commands

Use npm for package scripts.

```bash
npm install
npm run build
npm run typecheck
npm test
npm run dev -- --help
```

Run the CLI from source:

```bash
npm run dev -- --output-dir ./outputs "https://example.com/article"
```

Run the built CLI:

```bash
npm run build
node dist/cli.js --output-dir ./outputs "https://example.com/article"
```

Install Patchright Chromium when browser-based tests or manual browser fetching are needed:

```bash
npx patchright install chromium
```

## Coding Guidelines

### TypeScript and ESM

- Keep source files in TypeScript.
- Use ESM syntax (`import` / `export`).
- Include `.js` extensions in relative imports, even when importing TypeScript source files, because the project emits ESM JavaScript.
  - Good: `import { foo } from "./foo.js";`
  - Avoid: `import { foo } from "./foo";`
- Prefer explicit exported interfaces/types for cross-module data structures.
- Avoid `any`; use `unknown`, narrow errors, or define proper types.

### CLI behavior

- Keep `src/cli.ts` focused on argument parsing, validation, orchestration, and user-facing errors.
- Put reusable logic in modules under `src/` instead of embedding it in the CLI action.
- Validate option values early and return clear error messages.
- Preserve exit-code behavior:
  - processing failures should set `process.exitCode = 1`
  - invalid CLI usage/configuration should set `process.exitCode = 2`
- Write progress and diagnostics to `stderr`; generated files should be written to disk, not stdout.

### Fetching and scraping

- Respect the existing fetch-mode model: `auto`, `static`, `browser`, `stealth`.
- Prefer adding new behavior through `FetchHtmlOptions` and strategy modules instead of special-casing inside the pipeline.
- Keep static fetch fast and dependency-light.
- Browser/stealth behavior should remain configurable through CLI flags where appropriate.
- Do not add aggressive crawling, bypass logic, or rate-unfriendly defaults.
- Respect website terms, copyright, robots policies, and user-controlled rate limits.

### Markdown output

- Preserve frontmatter generated by `renderFrontmatter()` unless there is a deliberate schema change.
- Keep generated Markdown deterministic where practical.
- Avoid overwriting unrelated notes.
- Maintain source URL tracking so repeated runs can clean up previous notes for the same source.
- Keep asset paths local and portable.

### Site rules

- Site-specific extraction rules belong in `src/site-rules/*.toml`.
- Prefer narrow, domain-specific rules over broad selectors that could affect unrelated sites.
- Update `src/site-rules/README.md` when adding new rule conventions.
- Add regression fixtures/tests when changing extraction behavior for a known site.

### Tests

- Add or update Vitest tests for behavior changes.
- Prefer unit tests for parsing, output rendering, slicing, filename sanitization, and strategy fallback behavior.
- Avoid network-dependent tests by default.
- Use fixtures/mocks for HTML, fetch responses, and browser fetch functions.
- Before finishing substantive changes, run:

```bash
npm run typecheck
npm test
```

If a change touches the build or CLI entry point, also run:

```bash
npm run build
```

## Dependency Guidelines

- Do not add dependencies unless they materially simplify the implementation.
- Prefer built-in Node.js APIs when sufficient.
- Keep CLI startup reasonably fast.
- If adding a runtime dependency, update `package.json`, lockfile, and README/AGENTS notes when relevant.
- Use `npm` so `package-lock.json` stays consistent.

## Generated and Ignored Files

Do not commit generated or local-output directories unless explicitly requested.

Common generated/local paths:

```text
node_modules/
dist/
coverage/
outputs/
clippings/
.env
.DS_Store
```

The repository already ignores several of these in `.gitignore`. If new generated directories are introduced, update `.gitignore`.

## Documentation Guidelines

- `README.md` is user-facing. Keep it focused on installation, CLI usage, examples, and operational notes.
- `AGENTS.md` is maintainer/agent-facing. Keep it focused on architecture, commands, and contribution rules.
- When adding a CLI option, update README usage/options if it affects users.
- When changing internal architecture, update this file if it affects future agent work.

## Safety and Ethics

Feedloom is intended for personal archiving and legitimate content extraction workflows.

Agents should not introduce defaults or examples that encourage abusive scraping. Keep examples conservative and remind users to respect site terms, copyright, robots policies, authentication boundaries, and reasonable request rates.

## Before Handing Off

For non-trivial changes, verify as much as practical:

```bash
npm run typecheck
npm test
npm run build
```

Then summarize:

- files changed
- behavior changed
- tests run
- any known limitations or follow-up work
